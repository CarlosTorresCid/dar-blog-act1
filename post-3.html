Construyendo la app: retos y aprendizajes

Publicado el 5 de noviembre de 2025 Desarrollo

En esta entrada resumo las decisiones técnicas y los retos que estoy encontrando al construir la aplicación. El objetivo es mantener una base sólida y extensible, sin sobredimensionar el proyecto.

Arquitectura en capas (simple y mantenible)

  • Ingesta: importación de CSV bancarios y lectura OCR de tickets/facturas.
  • Procesado: limpieza/normalización, reglas de negocio y clasificación (reglas + IA ligera).
  • API/Servicio: operaciones sobre movimientos, categorías y métricas (REST).
  • Presentación: panel con KPIs y vistas de detalle: (inicialmente web simple; opción a app nativa después).

Tecnologías clave

  • Backend: Java (Spring Boot) para servicios REST y lógica de negocio.
  • Persistencia: PostgreSQL en desarrollo; SQLite como opción local “portátil”.
  • OCR: Tesseract (local) como base; interfaz preparada para alternar con un proveedor cloud si mejora calidad.
  • Clasificación: enfoque mixto:
    • Reglas deterministas (MCC, nombre comercio, patrones).
    • Modelo ligero entrenable con descripciones y feedback del usuario.
  • Frontend: HTML/CSS/JS minimalista para el panel inicial (tablas, filtros, gráficos sencillos).

Modelo de datos (mínimo necesario)

  • Movimiento: fecha, importe, signo, descripción, comercio, fuente (OCR/CSV), id categoría, etiquetas, aprobaciones.
  • Categoría/Subcategoría: jerarquía sencilla y editable.
  • Regla: condición → acción (asignar categoría, etiqueta, normalización).
  • Métricas: vistas/consultas para cálculo de totales, medias móviles y desviaciones.

Retos principales (y cómo los estoy abordando)

  1. OCR inconsistente: tickets arrugados o borrosos. Mitigación: preprocesado de imagen (contraste/umbral), plantillas por comercio repetido y confirmación asistida.
  2. Clasificación ambigua: descripciones diferentes para el mismo comercio. Mitigación: normalización de cadenas, alias por comercio y aprendizaje con correcciones del usuario.
  3. Privacidad: datos sensibles. Mitigación: almacenamiento local por defecto, exportación/importación y borrado completo con un clic.
  4. Experiencia de uso: demasiadas opciones abruman. Mitigación: panel con pocos KPIs útiles y acciones claras (revisar, confirmar, corregir).

Calidad y pruebas

  • Tests de servicio para reglas de categorización y normalización de importes/fechas.
  • Datos de demo (sintéticos) para validar el flujo completo sin exponer datos reales.
  • CI básica (GitHub Actions) para compilar y ejecutar tests en cada cambio relevante.

Roadmap inmediato

  1. MVP de importación CSV con normalización y reglas básicas.
  2. OCR de tickets frecuentes + interfaz de confirmación rápida.
  3. Panel con KPIs (ingresos, gasto por categoría, ahorro previsto) y alertas simples.
  4. Entrenamiento incremental del clasificador con feedback del usuario.

La prioridad es entregar valor desde el primer uso: registrar con pocos clics, entender en segundos y anticipar desvíos a tiempo. A partir de ahí, iterar sin romper lo que ya funciona.
